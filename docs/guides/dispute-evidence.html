Dispute Evidence Guide

Sending Evidence for your Dispute

Disputes occur when a customer initiates a chargeback with their bank or credit card company (check out our FAQ on disputes and chargebacks for more information). However, disputes can be overturned by the customer’s bank if they receive enough information showing that the chargeback was not justified. If this happens, we will refund any money that you lost during the dispute process (including any chargeback fees).

To help you win disputes, Stripe allows you to submit evidence to your customer’s bank. Moreover, we curate and format the evidence you submit so that you have the best chance of overturning a dispute. This guide will show you how to submit evidence to Stripe.

Using the Dashboard

There are two ways to submit evidence to Stripe. The first is to simply use our web interface. This is the recommended route since our interface will guide you through the submission process. The interface will ask for different pieces of evidence depending on the type of chargeback that your customer filed and will alert you of evidence types that are likely to win a dispute. It’s not required that you fill out every field to submit your dispute, but providing more relevant information will improve your chances of winning.

The second way to submit evidence is through the API, which the rest of this guide will outline in detail.

Structured Evidence

To submit structured evidence for a dispute via the API, you’ll need to update it by POSTing a hash to the evidence parameter of the dispute. The keys for the hash will be the evidence fields defined in the next section (you can also find them in the Full API Reference). The following example updates dispute evidence for the customer_email, service_date, and service_documentation fields.

POST /v1/charge/ch_5Q4BjL06oPWwho/dispute \
  -u SECRET_API_KEY: \
  -d "evidence[customer_email]=email@website.com" \
  -d "evidence[service_date]=2015-01-10" \
  -d "evidence[service_documentation]=file_5TjCEtQUx1si4y"

API Evidence Fields

This section provides all of the different evidence fields that can be used when updating a dispute. These fields fall into one of two categories: text evidence that take a string, or file evidence that take an uploaded file token. See the “File Evidence” section below for a complete description of the latter. You can find more detailed instructions on the evidence to provide depending on the type of dispute.

Text evidence fields

  • product_description: A description of the product or service which was sold.
  • customer_name: The name of the customer.
  • customer_email_address: The email address of the customer.
  • customer_purchase_ip: The IP address that the customer used when making the purchase. When we compile your evidence into a single document, this will be automatically expanded to include geographical data.
  • billing_address: The billing addess provided by the customer.
  • refund_policy_disclosure: An explanation of how and when the customer was shown your refund policy prior to purchase.
  • refund_refusal_explanation: A justification for why the customer is not entitled to a refund.
  • cancellation_policy_disclosure: An explanation of how and when the customer was shown your cancellation policy prior to purchase.
  • cancellation_rebuttal: A justification for why the customer’s subscription was not canceled.
  • shipping_address: The address to which a physical product was shipped. You should try to include as much complete address information as possible.
  • shipping_date: The date on which a physical product began its route to the shipping address, in a clear human-readable format.
  • shipping_carrier: The delivery service that shipped a physical product, such as Fedex, UPS, USPS, etc. If multiple carriers were used for this purchase, please separate them with commas.
  • shipping_tracking_number: The tracking number for a physical product, obtained from the delivery service. If multiple tracking numbers were generated for this purchase, please separate them with commas. When we compile your evidence into a single document, these tracking numbers will be expanded to include detailed delivery information from the carrier.
  • access_activity_log: Any server or activity logs showing proof that the customer accessed or downloaded the purchased digital product. This information should include IP addresses, corresponding timestamps, and any detailed recorded activity.
  • service_date: The date on which the customer received or began receiving the purchased service, in a clear human-readable format.
  • duplicate_charge_id: The Stripe ID for the prior charge which appears to be a duplicate of the disputed charge.
  • duplicate_charge_explanation: An explanation of the difference between the disputed charge and the prior charge that appears to be a duplicate.
  • uncategorized_text: Any additional evidence or statements.

File evidence fields (PDF, JPG, and PNG files are allowed)

  • refund_policy: Your refund policy, as shown to the customer.
  • cancellation_policy: Your subscription cancellation policy, as shown to the customer.
  • receipt: Any receipt or message sent to the customer notifying them of the charge. This field will be automatically filled with a Stripe generated email receipt if any such receipt was sent.
  • customer_signature: A relevant document or contract showing the customer’s signature.
  • shipping_documentation: Documentation showing proof that a product was shipped to the customer at the same address the customer provided to you. This could include a copy of the shipment receipt, shipping label, etc, and should show the full shipping address of the customer, if possible.
  • service_documentation: Documentation showing proof that a service was provided to the customer. This could include a copy of a signed contract, work order, or other form of written agreement.
  • duplicate_charge_documentation: Documentation for the prior charge that can uniquely identify the charge, such as a receipt, shipping label, work order, etc. This document should be paired with a similar document from the disputed payment that proves the two payments are separate.
  • customer_communication: Any communication with the customer that you feel is relevant to your case (for example emails proving that they received the product or service, or demonstrating their use of or satisfaction with the product or service).
  • uncategorized_file: Any additional evidence or statements.

If you’re only interested in submitting a single file or plaintext blob as evidence, use only the uncategorized_text or uncategorized_file fields. However, we recommend filling in as many of these fields as possible so we can optimize your evidence submission.

Fetching Evidence

The dispute object itself will have additional information about when your evidence is due and how many submissions have been attempted, as well as the submitted evidence.

GET /v1/charge/ch_5Q4BjL06oPWwho/dispute \
  -u SECRET_API_KEY:

The above returns the following in the response:

{
  object: "dispute"
  charge: "ch_5Q4BjL06oPWwho"
  evidence_details: {
    due_by: 1403047735,
    submission_count: 1
  },
  evidence: {
    customer_name: "Abraham Lincoln",
    customer_purchase_ip: "127.0.0.1",
    product_description: "Widget ABC, color: red",
    shipping_carrier: "Pony Express",
    shipping_tracking_number: "0123456789",
    uncategorized_text: "Additional notes and comments",
    uncategorized_file: "file_5TjCEtQUx1si4y"
  }
  ...
}

File evidence

For evidence fields that require uploaded files, you’ll need to use our uploads API.

First, POST a file as part of a multipart/form-data request to https://uploads.stripe.com/v1/files. For example:

POST https://uploads.stripe.com/v1/files \
  -u SECRET_API_KEY: \
  -F purpose=dispute_evidence \
  -F file=@/path/to/a/file

This will upload the file and return a file object:

{
  "id": "file_5TjCEtQUx1si4y",
  "object": "file",
  "created": 1403047735,
  "size": 4908
}

Once the upload is complete, you can use this file ID to attach the file to the correct evidence field:

POST /v1/charge/ch_5Q4BjL06oPWwho/dispute \
  -u SECRET_API_KEY: \
  -d evidence[uncategorized_file]=file_5TjCEtQUx1si4y
